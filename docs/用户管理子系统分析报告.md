# 用户管理子系统分析报告

## 1. 任务背景

Huobao Drama 已具备短剧生产链能力，包括项目创建、剧集创建、AI 改写、角色场景提取、分镜拆解、图片与视频生成、镜头合成和整集合并。原系统更接近单用户可信环境，缺少登录、用户、会话、成员关系和权限边界。本阶段目标是在现有系统上增加用户管理子系统，使制片人、内容编辑、技术运营能够在同一平台按权限协作。

## 2. 现状分析

### 2.1 当前系统的已有条件

后端采用 Hono、Drizzle ORM、SQLite，接口按业务模块拆分，数据库初始化集中在 `backend/src/db/index.ts`，表定义集中在 `backend/src/db/schema.ts`。前端采用 Nuxt 3，API 调用统一封装在 `frontend/app/composables/useApi.ts`，页面结构清晰，适合加入统一登录态和路由守卫。

### 2.2 当前系统的主要缺口

原系统没有用户表、会话表、项目成员表、认证中间件、全局角色判断、项目级权限判断、登录页面和前端路由守卫。所有接口默认可信调用，设置页、AI 配置、Agent 配置、Skills 维护等高风险功能缺少访问限制。

### 2.3 可行性结论

可行。系统已有统一 API 前缀 `/api/v1` 和清晰的数据库初始化逻辑，可以在不破坏旧数据的基础上新增用户、会话和项目成员表；前端也具备统一 API 封装和布局入口，适合实现登录态管理与界面权限控制。

## 3. 权限需求拆解

### 3.1 角色模型

系统采用两层角色。全局角色包括 `admin`、`operator`、`user`。项目级角色包括 `owner`、`producer`、`editor`、`viewer`。

### 3.2 权限边界

管理员可以管理用户和所有业务资源。技术运营可以维护 AI 服务、Agent、Skills 和素材生成问题。项目负责人拥有项目最高权限。制片人可以创建剧集、选择模型配置、推进图片视频生成、合成和合并。编辑可以录入原始内容、查看和修改 AI 改写结果、维护角色、场景和分镜。只读成员只能查看项目。

### 3.3 为什么要做“项目级成员关系”

如果只使用全局角色，所有编辑或制片人会互相访问项目，无法体现项目协作边界。项目级成员关系可以把用户和具体短剧项目绑定起来，支持同一平台内多项目、多职责协作。

## 4. 总体实现方案

### 4.1 方案总览

后端新增用户、会话、项目成员表；新增认证服务、认证中间件、认证接口和用户管理接口；原有业务接口按全局角色和项目角色做授权。前端新增登录页、登录态 composable、路由守卫、用户管理页和项目成员管理区域。

### 4.2 认证方式选择

采用服务端 Session + HttpOnly Cookie。后端保存 token hash，前端只通过 Cookie 自动携带凭证，登出时服务端吊销会话。

### 4.3 鉴权模式选择

采用 RBAC + 项目成员关系。系统级资源通过全局角色控制，项目内资源通过 `drama_members` 判断用户是否属于项目以及在项目中的角色。

## 5. 数据库设计

### 5.1 建议新增的数据表

已新增 `users`、`user_sessions`、`drama_members`、`audit_logs`。其中 `users` 保存账号、密码哈希、全局角色和状态；`user_sessions` 保存会话 token hash、过期时间和吊销状态；`drama_members` 保存项目成员关系和项目角色；`audit_logs` 预留审计能力。

### 5.2 数据兼容性分析

新增表使用 `CREATE TABLE IF NOT EXISTS`，不修改原有业务表字段。旧项目仍可由管理员和技术运营查看；新建项目会自动把创建者写入 `drama_members` 并设为 `owner`。

## 6. 后端实现设计

### 6.1 建议新增与修改的文件

新增 `backend/src/services/auth.ts`、`backend/src/middleware/auth.ts`、`backend/src/routes/auth.ts`、`backend/src/routes/users.ts`。修改 `db/index.ts`、`db/schema.ts`、`index.ts` 和多个业务路由。

### 6.2 认证接口设计

提供 `POST /api/v1/auth/login`、`POST /api/v1/auth/logout`、`GET /api/v1/auth/me`、`POST /api/v1/auth/bootstrap`。

### 6.3 中间件设计

`requireAuth` 校验 Cookie 中的会话；`requireGlobalRole` 校验全局角色；`requireExistingDramaAccess` 根据项目 ID 和项目角色校验项目权限。

### 6.4 路由授权方案

用户管理仅管理员可写；AI 配置读取允许项目创建流程使用，写入和测试限制为管理员或技术运营；Agent 配置、Skills、音色同步限制为管理员或技术运营；项目资源根据成员关系授权。

### 6.6 安全细节

密码使用 `crypto.scrypt` 加盐哈希；session 数据库存储 token hash 而不是明文 token；Cookie 设置 HttpOnly；删除用户采用停用和软删除策略；静态资源暂不做鉴权，后续可增强。

## 7. 前端实现设计

### 7.1 需要新增和修改的文件

新增 `useAuth.ts`、`middleware/auth.global.ts`、`pages/login.vue`、`pages/users.vue`。修改 `useApi.ts`、`layouts/default.vue`、`pages/drama/[id]/index.vue`。

### 7.2 登录态管理方案

前端通过 `useAuth` 保存当前用户、登录状态和角色判断。API 请求统一使用 `credentials: 'include'` 携带 Cookie，401 时跳转登录页。

### 7.3 页面级改造

登录页用于账号登录；默认布局展示当前用户和退出按钮；设置页仅 admin/operator 可见；用户管理页仅 admin 可见；项目详情页增加成员管理。

## 8. 接口设计

认证接口、用户管理接口、项目成员接口和原业务接口共同构成用户管理子系统。项目成员接口包括 `GET /dramas/:id/members`、`POST /dramas/:id/members`、`PUT /dramas/:id/members/:userId`、`DELETE /dramas/:id/members/:userId`。

## 9. 关键实现步骤

第一步新增数据库表；第二步实现认证服务和中间件；第三步实现登录、登出、当前用户和用户管理接口；第四步改造项目与业务接口权限；第五步实现前端登录态、登录页、用户管理和成员管理；第六步执行测试并整理报告。

## 10. 风险与难点分析

主要风险是资源归属反查复杂，例如分镜需要通过剧集反查项目；前端隐藏按钮不能替代后端授权；旧数据缺少项目 owner；Webhook 和静态文件需要保留外部访问能力。当前实现通过后端强制鉴权、管理员兼容旧项目、保留公开静态资源来降低风险。
